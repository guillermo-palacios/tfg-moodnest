\section{Pruebas y Validación del Sistema}

La fase de pruebas constituye un pilar fundamental en el ciclo de vida del desarrollo de software, garantizando que el producto final no solo cumple con las especificaciones funcionales y no funcionales analizadas, sino que también ofrece un nivel óptimo de calidad, seguridad y tolerancia a fallos. En este capítulo se detallan las estrategias de validación aplicadas sobre el ecosistema de MoodNest, abarcando desde la integración de la API REST hasta la experiencia de usuario final.

\subsection{Pruebas Unitarias (Backend)}

Siguiendo los principios de la pirámide de pruebas de la ingeniería de software, la primera línea de defensa para garantizar la calidad del código ha sido la automatización de pruebas unitarias sobre el \textit{backend}. El objetivo de esta fase es aislar y verificar la corrección algorítmica de la lógica de negocio pura, independientemente de la base de datos, la red o la interfaz visual.

Para el ecosistema Spring Boot, se ha utilizado el marco de pruebas \textbf{JUnit 5} \cite{bib:junit} en combinación con la librería \textbf{Mockito} \cite{bib:mockito}. Esta última resulta fundamental, ya que permite crear objetos simulados de los repositorios de datos, evitando que las pruebas dependan de una conexión real a MongoDB y garantizando que se ejecuten en milisegundos.

La cobertura de estas pruebas se ha enfocado de manera estratégica en el paquete \texttt{services}, dada su criticidad arquitectónica. Se ha diseñado una batería de evaluación sobre las clases \texttt{UsuarioService}, \texttt{RegistroDiarioService} y \texttt{EtiquetaService}. Entre los escenarios evaluados destacan:
\begin{itemize}
    \item \textbf{Validación de Restricciones Temporales:} Se ha comprobado mediante aserciones que el servicio intercepta y bloquea la creación de registros asociados a fechas futuras.
    \item \textbf{Aislamiento de Entidades y Permisos:} Se ha verificado que el sistema procesa adecuadamente los identificadores de usuario, bloqueando intentos de mutación sobre registros o etiquetas pertenecientes a cuentas ajenas.
    \item \textbf{Integridad Histórica (Borrado Lógico):} Se ha testeado que la eliminación de etiquetas ejecuta una mutación del estado lógico en lugar de una destrucción física en la base de datos.
    \item \textbf{Operaciones Críticas y Cascada:} Se ha garantizado la fiabilidad del algoritmo de cálculo de rachas de días consecutivos, así como el borrado en cascada del historial al eliminar una cuenta de usuario.
\end{itemize}

Además de validar el comportamiento esperado del sistema, las pruebas unitarias han permitido identificar defectos funcionales durante el desarrollo. Un caso especialmente relevante se produjo en la gestión de las rachas de actividad. Se diseñó una prueba específica para verificar que, al eliminar un registro diario perteneciente a una secuencia consecutiva de días, el sistema recalculara automáticamente la racha del usuario. La ejecución inicial de dicha prueba produjo un fallo, evidenciando que el método encargado de actualizar la racha no era invocado tras la eliminación del registro.

\subsection{Pruebas de Integración y API REST (Backend)}

Antes de conectar la interfaz visual, se procedió a la validación exhaustiva de la capa de servicios a través de la red para garantizar la solidez de la comunicación y la seguridad perimetral. Para ello, se empleó la herramienta \textbf{Postman} \cite{bib:postman}, simulando las peticiones HTTP que el cliente web realizaría en un entorno real. El plan de pruebas sobre la API se centró en dos vectores críticos, la autenticación y la seguridad perimetral, descritos en detalle a continuación.

\subsubsection{Autenticación y Generación de Credenciales}

El primer hito de las pruebas de integración consistió en certificar el correcto funcionamiento del módulo de seguridad y la emisión de tokens. Como se observa en la Figura \ref{fig:postman_registro}, al realizar una petición legítima de registro al \textit{endpoint} público, el sistema procesa las credenciales, encripta la contraseña en la base de datos y genera exitosamente un \textit{JSON Web Token} (JWT) firmado. Este token se devuelve al cliente para que sea utilizado como salvoconducto (\textit{Bearer Token}) en todas las operaciones posteriores.

    \caption{Creación de usuario y emisión exitosa del token JWT (Código 200 OK).}
    \label{fig:postman_registro}
\end{figure}

\subsubsection{Operaciones Transaccionales y Seguridad Perimetral}

Una vez garantizada la emisión de credenciales, se evaluó el comportamiento de los \textit{endpoints} privados. Cuando las peticiones adjuntan el token JWT válido en la cabecera HTTP, las operaciones (como la creación de un registro diario) son procesadas exitosamente (ver Figura \ref{fig:postman_200}).

Por el contrario, para certificar la robustez del escudo arquitectónico, se lanzaron peticiones intencionadas a rutas privadas omitiendo deliberadamente el token JWT.

Como ilustra la Figura \ref{fig:postman_403}, el filtro de seguridad de Spring actuó de forma implacable ante la ausencia de credenciales, denegando la comunicación en la capa de red y devolviendo un estado \texttt{403 Forbidden} (Prohibido). Esto garantiza que la cadena de filtros intercepta a los usuarios no autenticados, manteniendo la lógica de negocio subyacente y la privacidad de los datos totalmente inaccesibles para clientes anónimos o maliciosos.

\subsection{Validación Funcional (Casos de Uso)}

Una vez consolidada la API REST, se ejecutó un plan de pruebas de caja negra sobre el \textit{frontend} utilizando la técnica de recorridos funcionales. Se verificaron uno a uno los flujos principales y alternativos de los Casos de Uso definidos en el análisis.

Gran parte de la validación resultó exitosa a la primera iteración. Destacan especialmente el \hyperref[tab:esp_cu6]{CU6 (Crear Registro)}, donde el sistema bloqueó exitosamente los intentos de registrar jornadas en fechas futuras; el \hyperref[tab:esp_cu8]{CU8 (Gestionar Etiquetas)}, donde se comprobó que el borrado lógico ocultaba la etiqueta de los nuevos menús pero preservaba intacto el historial de los registros pasados; y los \hyperref[tab:esp_cu11]{CU11 a CU13 (Estadísticas)}, cuyos algoritmos procesaron adecuadamente los huecos cronológicos sin distorsionar la gráfica evolutiva.

\subsubsection{Incidencias, Adaptaciones y Desviaciones Detectadas}

Durante la ejecución del plan de pruebas, la interacción real con la interfaz puso de manifiesto áreas de mejora en la Experiencia de Usuario (UX), lo que motivó la refactorización de ciertos flujos respecto a la especificación original:

\begin{itemize}
    \item \textbf{CU1 (Registrar Usuario):} El análisis inicial estipulaba que, tras un registro exitoso, el sistema debía redirigir al usuario a la pantalla de inicio de sesión. Durante las pruebas, se consideró que esto añadía fricción innecesaria. Se modificó el comportamiento para que el servidor generase y devolviese un token JWT inmediatamente tras el registro, redirigiendo al usuario directamente al Panel Principal (\textit{Dashboard}).

    \item \textbf{CU4 (Cerrar Sesión):} Se detectó una desviación respecto al flujo planificado. Originalmente se diseñó un cuadro de diálogo de confirmación previa a la salida. Sin embargo, en la implementación inicialmente se creó un modelo de acción directa (\textit{One-Click Logout}), cerrando la sesión instantáneamente y redirigiendo al usuario a la pantalla de inicio de sesión, por lo que tras la revisión del flujo se detectó la contradicción con el CU y se corrigió.

    \item \textbf{CU5 (Eliminar Cuenta):} En este caso sucedió algo parecido al caso anterior, ya que la cuenta se eliminaba por completo al pulsar el botón sin solicitar las confirmaciones necesarias, por lo que tras detectar el problema se implementó el flujo correcto para la eliminación de la cuenta.
\end{itemize}

\subsection{Validación de Requisitos No Funcionales (RNF)}

La evaluación de los atributos de calidad arquitectónica (Rendimiento, Usabilidad, Seguridad y Portabilidad) determinó el grado de madurez técnica del sistema. A continuación se exponen los resultados de las pruebas realizadas sobre los RNF críticos.

\subsubsection{Rendimiento y Tolerancia a Fallos}

\begin{itemize}
    \item \textbf{RNF2.1 y RNF2.2 (Tiempos de Respuesta y Renderizado):} Mediante las herramientas de desarrollador del navegador (\textit{DevTools}), se monitorizó el tráfico de red (Figura \ref{fig:prueba_red}). Las respuestas de la API REST mantuvieron promedios excelentes de entre 12 y 107 milisegundos, manteniendose inferior a los 2 y 3 segundos pedidos por los \hyperref[tab:rnf_total]{RNF}.

    \item \textbf{RNF2.4 (Caída de Base de Datos y Timeout):} Durante las pruebas de estrés, se forzó la detención del contenedor de MongoDB. Inicialmente, este escenario provocó el bloqueo indefinido del cliente web. Por ello, para solventar este defecto, se reconfiguró el sistema en el \textit{frontend}, donde se inyectó un \texttt{timeout} explícito de 5000ms en el cliente Axios. Tras la corrección, la caída del motor de base de datos se tradujo en una intercepción controlada del error y el lanzamiento inmediato de una notificación de fallo de red al usuario, evitando la congelación de la pantalla.
\end{itemize}

\subsubsection{Seguridad, Usabilidad y Compatibilidad}

\begin{itemize}
    \item \textbf{Protección Criptográfica (RNF4.1 y RNF4.2):} Se validó de forma satisfactoria el almacenamiento cifrado mediante auditoría de la base de datos, constatando que el \textit{backend} genera firmas irreversibles de 60 caracteres mediante \texttt{BCrypt} \cite{bib:bcrypt}. Adicionalmente, la manipulación o borrado manual del token JWT en el almacenamiento local del navegador provocó la expulsión inmediata y automática del usuario de las rutas privadas gracias al escudo lógico del componente \texttt{<ProtectedRoute>}.
    \item \textbf{Cifrado de Tráfico (RNF4.4):} Durante el desarrollo local, el sistema operó sobre el protocolo HTTP sin cifrado para agilizar la depuración. El requisito de seguridad de la capa de transporte se cumplirá de forma delegada en la fase de despliegue a Producción, donde se configurará un proxy inverso respaldado por certificados SSL/TLS para forzar la encriptación HTTPS.
    \item \textbf{Usabilidad y Navegadores (RNF1.2 y RNF3.1):} La compatibilidad Multi-Navegador se certificó ejecutando la aplicación de forma paralela en navegadores basados en motores Chromium (Google Chrome) y motores Gecko (Mozilla Firefox bajo SO Ubuntu Linux), sin encontrar discrepancias de diseño. Asimismo, la simulación de resoluciones móviles confirmó el comportamiento responsivo del componente \texttt{Layout}, ocultando el menú superior en favor de una barra de navegación inferior orientada al pulgar.
    \item \textbf{Eficiencia de Navegación (RNF3.2 y RNF3.3):} Se superó con éxito la métrica de accesibilidad transaccional: un usuario puede crear un registro diario con un máximo de dos interacciones (clic en el logo de MoodNest y clic en el botón central del Dashboard). A su vez, el sistema centralizado de notificaciones (\texttt{react-hot-toast}) \cite{bib:react_hot_toast} proveyó tiempos de \textit{feedback} visual muy inferiores al límite de medio segundo exigido.
\end{itemize}

La superación de este marco de pruebas certifica que MoodNest cuenta con una implementación madura, tolerante a fallos y preparada funcionalmente para su entrada en un entorno de producción o uso real.
